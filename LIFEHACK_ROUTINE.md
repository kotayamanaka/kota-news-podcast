# テックラジオ — 番組差分ルーチン（SHOW=lifehack）

> 制作の本体手順は **RADIO_BASE.md** に従う。このファイルは**テーマ選定と番組の狙い**だけを定義する。
> scheduled-task：`lifehack-radio`（毎朝3:30頃。全番組を6時までに仕上げる方針）。配信フィード：`lifehack.xml`（番組名・方針は変わったが購読維持のためフィードIDは lifehack のまま）。
> 2026-06-20「ライフハックラジオ」→「テックラジオ」へ改名。**2026-06-28 方針転換（kota 指示）**：「他人のアプリ・サービスを1つ紹介する」路線を廃止。**AIエージェント・バイブコーディングで、自分で仕事や生活を便利にするアプリ／仕組みを作る・考える**ための、アイデア・実例・テクニックを扱う番組へ。

## この番組の狙い

kota が **AIエージェント・バイブコーディングを使って、自分の仕事や生活を便利にするアプリ・仕組みを作る／考える** ための、アイデア・実例・テクニックを1日1つ深掘りする。

- 核は「**他人が作ったサービスを使う**」ではなく「**自分で作る・仕組み化する**」側。いまは非エンジニアでもバイブコーディング（AIにコードを書かせる開発）で、自分専用のアプリや自動化を作れる時代。その発想・事例・コツを集めて届ける。
- 想定する3系統のネタ（STEP B 参照）：①**こういう仕組み／アプリを作ると便利**というアイデアと作り方の発想、②**非エンジニアが実際にバイブコーディングで作った事例**とその工夫・ハマりどころ、③**最新のAI活用テクニック・エージェント/ハーネス設計**の解説。
- ゴールは「kota（や同じく非エンジニアの作り手）が、**今日これを真似して自分でも作れる・設計できる**」粒度で届けること。単に「便利なサービスがありますよ」で終わる紹介はしない。
- ⚠️ 旧路線（実在ツール・Webサービスを1つ紹介して「使ってみよう」）は廃止。**ツールは手段として登場してよいが、主役は「それを使って何をどう作る／自動化する／設計するか」**。

## STEP B. テーマ選定（毎日1つ）

以下3系統を巡回・ミックスして1つ選ぶ（その日いちばん鮮度・実用度が高いものを優先）。

1. **自作の仕組み・アプリのアイデア**：仕事や生活の不便を、AIエージェント／バイブコーディング／自動化でどう解消できるかを1つ。発想だけでなく「どう組むか」の設計まで踏み込む。
   - 例：スキャンしたPDFを定期実行で自動リネーム＆フォルダ振り分けする仕組み（kota 実装済み）／自作タスク管理アプリに足すと効くおすすめ機能・使い方／メール・カレンダー・家計簿を横断する個人ダッシュボード／定期バッチで情報を集めて要約・通知する仕組み 等。
2. **非エンジニアのバイブコーディング実例・工夫**：実在の個人（X・ブログ・note・GitHub・YouTube 等）が、Claude Code / Cursor / Replit / GAS / v0 などで自分用アプリや自動化を作った事例を調べ、何を・どう作り・どこでつまずき・どう乗り越えたかをまとめる。出典必須。
3. **最新AI活用テクニック／エージェント・ハーネス設計**：作る力そのものを上げるネタ。例：Claude Code の loop コマンド・サブエージェント並列・MCP連携・スケジュール実行、効くプロンプト/ハーネスの設計原則、バイブコーディングの進め方（仕様の渡し方・反復の回し方・つまずきの抜け方）、自動化パイプラインの組み方 等。

- **優先順位**：①再現性が高く今日から作り始められる ②kota の作り手志向（自動化・仕組み化・個人開発好き、AIヘビーユーザー）に効く ③新しさ・意外性がある。
- ⚠️ **「kota がまだ知らない／やっていない」鮮度を重視**。誰でも知っている定番（「ChatGPTで議事録要約」級）や、kota が既にやっている運用は退屈。マイナーな設計パターン・最近出た機能・尖った個人事例・意外な応用を狙う。
- `data/lifehack_history.json` の既出と被らないこと。

## STEP D. リサーチの観点（RADIO_BASE のリサーチに加えて）

1. **何を作る／どう便利になるか**（解決する不便・ビフォーアフター）
2. **具体的な作り方・設計**（使う道具＝AIエージェント/言語/サービス、構成、手順の骨子。非エンジニアでも追える粒度で）
3. **実在の作り手・出典**（個人ブログ／X／GitHub／公式ドキュメント。事例として誰がどう作ったか）
4. **始め方の最初の一歩**（今日、最初に何を作る/試すか）
5. **落とし穴・限界**（つまずきやすい所、コスト、セキュリティ・プライバシー・規約・保守の手間など正直に）

## STEP F. 台本の作り（RADIO_BASE の構成に沿って）

- 導入：いま誰もが困りがちな不便、または「非エンジニアでも自分で作れる時代になった」という事実ベースのトレンド。
- まず「今日のテーマ＝何を作る／どんなテクニックか」を一言で。
- トピック3つは系統に応じて、**自分で再現できる粒度**で。例：①どんな不便をどう解決するか／正体 ②具体的な作り方・設計・実例（誰がどう作ったか）③始め方・注意点と限界。便利さだけでなく**つまずき所・限界も正直に**。
- おわりに：**「まず最初の一歩」**を1つ具体的に（今日これを作り始める／この設計を試す）。
- ツール名・言語・サービス名・コストは具体的に（URLは `_meta` の説明欄に入れる）。

### 冒頭・結びの定型
- 冒頭：「こんにちは。テックラジオ、今日のテーマは『○○』です。」
- 結び：「以上、今日のテックラジオでした。まずは○○から、作ってみてください。」（テクニック解説回は「試してみてください」でも可）
