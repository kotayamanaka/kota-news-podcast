# サブカルラジオ — 毎日の生成ルーチン

> このファイルは毎朝（5:10頃。全番組を6時までに仕上げる方針）の scheduled-task（`katareru-radio`）が読んで **そのまま実行する** 手順書。SHOW=katareru（旧「語れるラジオ」。2026-06-19 に kota 指示で「サブカルラジオ」へ改名。フィードIDは購読維持のため katareru のまま）。
> 目的：いま話題の **サブカルチャー**（ネットのミーム・流行・アニメ・漫画・ゲーム・VTuber・キャラ・ネット文化・新しい商品やブームなど）を **1つ** 選び、他人に語れるレベルまで深掘りして、約9〜11分の音声（mp3）にして Podcast 配信する。
> ※扱うのは**サブカル寄りの話題**に絞る（経済概念・国際情勢・お堅いビジネス用語などは扱わない。それらはビジネスラジオの領分）。「名前は聞くけど中身を説明できる人が少ない」サブカル現象が主役。
> 作業ディレクトリ：`C:\Users\kota\Documents\Claude\kota-news-podcast`
> 番組ID：`katareru`（朝の深掘りニュースとは **別フィード**。配信フィードはリポジトリ直下 `katareru.xml`。音声は `katareru/episodes/`）

この番組は note シリーズ「『○○』について語れるようになりたい」の音声版。記事制作の方法論は Vault の `Projects/語れるようになりたい/article_playbook.md` が正本。本ルーチンはそれを音声向けに要約・適応したもの。**矛盾したら article_playbook を優先**。

## 全体フロー

```
話題の「実はよく知らない単語」を横断収集 → 1つ選定（既出除外）→ 深掘りリサーチ
 → 音声台本（導入→まず○○とは→実はこうなってる×3→おわりに・約2000字）
 → 読みカナ化 → edge-tts → mp3 → publish.py（--show katareru）→ git push → history 追記
```

---

## STEP 1. 日付の確定

- 今日の日付を `YYYY-MM-DD`（JST）で確定。以降 `<DATE>` と書く。

## STEP 2. テーマ候補の横断収集

WebSearch を複数本投げ、「**話題には上がってるけど、実はよくわからない単語・人物・現象**」の候補を10〜15個リストアップする。例：「もっちゅりん」のような、SNS・ニュース・街中で見聞きするが中身を説明できる人が少ないもの。

カバーする領域（**サブカル寄りに絞る**）：
- SNS・ネットでバズってる単語／ミーム／キャラ／VTuber／配信者／コンテンツ
- 日本のエンタメ・カルチャー（アニメ・漫画・ゲーム・音楽・映画・お笑い・特撮・アイドル）
- ネット文化・スラング・流行りの遊び・バズった現象
- 新しい商品・ブーム・サブカル的な流行（食・グッズ・キャラビジネス等）
- ※テック/AIの専門用語・経済/金融/国際情勢などお堅い概念は扱わない（ビジネスラジオの領分）。サブカルとして語れるものを優先。

選定の必須条件（article_playbook より）：
- **「話題には上がってる」が必須**。誰も話題にしてないマイナー単語は弾く（＝Wikipedia の領分）。
- 「名前は聞くけど、中身を語れる人が少ない」＝解説の価値が高いものを優先（＝「実は」度）。

## STEP 3. 既出の除外

- `data/katareru_history.json` を読み、過去に取り上げたテーマと **被る候補を除外**する。

## STEP 4. 1つに絞る（スコアリング）

各候補を採点し、**最上位の1つ**を選ぶ：

| 観点 | 中身 |
|---|---|
| 話題性（最重要・必須） | いま世間で本当に話題か。ゼロなら弾く |
| 「実は」度 | 名前は聞くが中身を語れる人が少ない＝解説の価値 |
| 深掘り余地 | 「実はこうなってる」が3つ立つだけの背景・ヒストリーがあるか |
| 鮮度 | 直近で盛り上がっているか |
| 未出 | history と被らない |

選んだ1つと「なぜ選んだか」を1行、最後の報告に残す。

## STEP 5. 深掘りリサーチ（並列サブエージェント）

article_playbook §3 準拠。general-purpose サブエージェントを **3〜5体並列**で投げ、観点を分ける：

| 観点 | 内容 |
|---|---|
| ①本体の正体・基本情報 | 何者か・正式名称・構成要素・公式設定・基本データ |
| ②ヒストリー・時系列 | 初出から現在までの節目・年表 |
| ③爆発のきっかけ・拡散構造 | バズ起点・拡散経路・著名人言及・数字 |
| ④現在の展開全体像 | コラボ・グッズ・メディア出演・海外展開・ランキング |
| ⑤制作元・関連人物 | 制作会社・所属・パートナー（薄くてよい・読み物として効くエピソードだけ） |

各サブエージェントへの指示に必ず入れる（article_playbook §3 テンプレ）：
- 仮説・推測は一切書かない。「〜と言われている」「〜だろう」禁止
- 出典 URL を全件付記。日本語と英語の両方で検索
- 内部情報・噂は除外。公式発表・取材記事・本人インタビューに限定
- 確認できなかった項目は「未確認」と明記
- 最後に「特に面白い小ネタ Top 5」を列挙

### ファクトチェック（article_playbook §3 厳守）

- 人名・グループ名・固有名詞の表記は公式 or Wikipedia で確認
- 「〜と言われている」系の通説は一次ソースで裏取り。取れないなら書かない
- 数値は公式リリース・大手メディアで裏取れたものだけ。考察ブログ・Wikipedia 単独ソースの数値は使わない
- ネタバレ配慮（ゲーム・ドラマ・映画）

## STEP 5.5. 素朴疑問チェック（必須・当たり前回避）

リサーチ完了時に「**普通の人がこのテーマで浮かべる素朴な疑問**」を3〜5個書き出す（AIは整理された切り口に流れて素朴な疑問を取りこぼす）。例：「なんでこんな名前なの？」「誰が作ってるの？」「なんで急に流行ったの？」「結局どこがすごいの？」。各疑問が台本の中で読者（聴き手）に届くか点検し、届かないものはトピック化 or 「まず○○とは」に織り込む。

## STEP 6. 音声台本化（約2000字＝約6〜8分）

article_playbook §1 の構成テンプレを **音声向けに適応**する：

```
導入（話題性フック）
 → まず、○○とは（正体・現在地の数字を最小限）＋「実はこうなってる、を3つ共有します」予告
 → トピック1：「実は〜」その1
 → トピック2：「実は〜」その2
 → トピック3：「実は〜」その3
 → おわりに（最後に小ネタを1個・余韻で締める）
```

### 音声版で特に守ること

- **導入は捏造しない**：note 記事は冒頭に kota の実体験エピソードを置くが、**これは kota 本人が書く前提**（article_playbook §8）。自動生成の音声では kota の体験を捏造してはいけない。代わりに **事実ベースの話題性フック**で入る（「最近 SNS で『○○』って見かけませんか」「いま話題の『○○』、名前は聞くけど説明できる人、意外と少ないんですよね」など、事実として確認できる範囲で）。
- **トピックは3つが基準**。タイトルは具体的に（「実は5年熟成されてる」のように何が"実は"か一目で分かる形）。「○○の系譜」「○○の構造」のような抽象タイトルは弱い。
- **事実・固有名詞・数字で読ませる**。固有名詞の密度が濃さ。

### やらない（article_playbook §2・kota フィードバックの蓄積）

- ❌ 抽象化・補助線・系譜論（「○○が△△を売ってるなら、□□は〜を売ってる」型の AI 視点まとめ）
- ❌ 「○○の末っ子」「○○の構造的な面白さ」のような一段抽象化フレーズで核を作る
- ❌ 「明日誰かに振れる一文」のような締めの強い断定キャッチコピー
- ❌ 末尾の独立した小ネタ集（本文＝台本の流れに溶かし込む）
- ❌ 制作元の会社情報を厚く（沿革・社員数・事業モデル）／受託実績・受賞歴の羅列
- ❌ 題材に合わないフレーミングを勝手に当てる（題材の素直な論理に寄せる）
- ❌ 数値フックの空振り（事実が複数イベントで蓄積されているときだけ「○年熟成」等を使う。1点しか事実がないのに数値で括らない）

### 文体

- ですます調・会話寄り（「〜なんですよね」「〜という話でした」「実はこうなってて」「ちなみに」「地味に珍しい」）。
- 「〜と感じます」「〜なのです」のような演説調は避ける。
- 凝った語りフックは不要。素直に分かりやすく。

## STEP 7. テキスト整形＋読み正規化

台本を1つのテキストにまとめる。冒頭・結びの定型：
- オープニング：「こんにちは。サブカルラジオ、今日のテーマは『○○』です。」
- クロージング：「以上、今日の『○○』でした。これで明日、ちょっと語れますね。」

読み正規化（ROUTINE.md STEP 7 と同じ規律）：
- 英略語はカタカナ読みに（AI→エーアイ 等）。記号を言葉に（%→パーセント、$→ドル、〜→から）。
- 数字・日付・金額・件数は数字のまま残す（edge-tts が正しく読む）。

台本テキストは `scripts/_today_katareru.txt`（作業用・コミットしない）に保存。

## STEP 7.5. 読み上げ用カナ台本（Claude本人がふりがなを振る・必須）

ROUTINE.md STEP 7.5 と同じ。**ふりがなの主役は Claude（あなた）**。形態素解析器は読みを外すので機械任せにしない。台本の漢字を文脈に合う正しい読みでカナ化し、`scripts/_today_katareru.txt` を上書き保存する。

- **固有名詞・人名・地名・作品名・キャラ名は読みを確認して正しいカナに**（不確かなら WebSearch。語れるラジオは新語・固有名詞が主役なので特に重要）。
- 数字・日付・金額は数字のまま残す。
- 毎回出る固有名詞・定番語は `data/yomi_dict.json` に追記（朝ニュースと **共有**。kota に聞かず追記してよい）。
- 仕上げの保険：`python scripts/yomi_check.py scripts/_today_katareru.txt` で漢字残りを確認可。

## STEP 8. 音声合成

```
python scripts/synth.py scripts/_today_katareru.txt katareru/episodes/<DATE>.mp3
```

（声 = Nanami 固定。synth.py が data/yomi_dict.json を自動適用。朝ニュースと共通。）

## STEP 9. 配信（feed 更新）

**日本語をコマンドライン引数で渡さない**（Windows で文字化け）。タイトル・概要は必ず **ファイル経由**：

1. `scripts/_meta_katareru.json` を UTF-8 で書く：
   ```json
   { "title": "「○○」について語れるようになりたい", "desc": "<1〜2文の内容紹介>" }
   ```
2. 引数なしで publish（`--show katareru` を必ず付ける。title/desc は _meta_katareru.json から読まれる）：
   ```
   python scripts/publish.py --show katareru katareru/episodes/<DATE>.mp3 <DATE>
   ```

- これで `katareru/episodes/<DATE>.mp3` 配置、`katareru.xml` 再生成、直近30本に剪定、`data/katareru_episodes.json` 更新まで完了。
- ⚠️ `--show katareru` を忘れると朝ニュース側の feed.xml を壊す。**必ず付ける**。

## STEP 10. git push

```
git add -A
git commit -m "katareru <DATE>: <テーマ名>"
git push
```

- push 後、数分で `https://kotayamanaka.github.io/kota-news-podcast/katareru.xml` に反映され、Pod アプリが自動取得する。

## STEP 11. history.json 追記

- 今日のテーマを `data/katareru_history.json` に追記（date / theme / keywords）。重複回避のため必須。STEP 10 の前に追記して同じ commit に含める。

## 失敗時

- どこかで詰まったら、その時点までで分かったことをログに残して中断。不完全・壊れた音声は配信しない（git push しない）。
- edge-tts が落ちたら時間を置いて1回リトライ。

## 完了報告

- 選んだテーマと「なぜ選んだか」を1行で。
- トピック3つの見出しを列挙。
- 音声の長さ（分秒）と feed 反映を報告。
