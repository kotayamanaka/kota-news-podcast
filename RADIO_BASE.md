## STEP A. 日付確定
- 今日の日付を `YYYY-MM-DD`（JST）で確定。以降 `<DATE>`。

## STEP B. テーマ選定
- **`<SHOW>_ROUTINE.md` の選定ロジックに従って、今日の1テーマを決める**（番組ごとに違う）。

## STEP C. 既出除外
- `data/<SHOW>_history.json` を読み、過去に取り上げたテーマと被るものを除外。

## STEP D. 深掘りリサーチ（並列サブエージェント・article_playbook §3 準拠）
- general-purpose サブエージェントを**3〜5体並列**で投げ、観点を分ける（正体/基本・ヒストリー・きっかけ/数字・現在の展開・関連人物 等。番組により観点は調整）。
- 各エージェントに必ず指示：**仮説・推測禁止／出典URL全件／日英両方で検索／公式・取材・本人発言に限定／未確認は「未確認」と明記／面白い小ネタTop5**。
- ファクトチェック：人名・固有名詞の表記は公式orWikipedia、数値は公式・大手メディアで裏取れたものだけ。取れない数値は使わない。

## STEP E. 素朴疑問チェック（必須）
- 「普通の人がこのテーマで浮かべる素朴な疑問」を3〜5個書き出し、台本で答えられているか点検。当たり前の問いは避け、非自明な「なぜ」に寄せる。

## STEP F. 音声台本化（約9〜11分を目標）

> **長さの目安：1回 約9〜11分。読み上げ用カナ台本で約3300〜3900字**（句読点・間で前後する。コンマ多めの語り口だと同じ字数でも長くなる）。**生成後に必ず長さを確認し、7分未満なら深掘りを足して作り直す**。各トピックを下の「深掘りの作法」でしっかり掘れば自然とこの分量になる（1トピック約1000〜1200字）。

### 深掘りの作法（最重要・kota指示 2026-06-18）

**表面の答えで止めない。「なぜ」を一段で終わらせず、連鎖で掘る。** とくに次の2つの問いに必ず踏み込む：

1. **「なぜ、それが可能なのか」**（その仕組み・強み・現象が成り立っている、裏のメカニズム）
2. **「なぜ、他はできない／やらないのか」**（他社・他制度・普通の人と比べて、何が違うから真似できないのか。参入障壁・経路依存・自己強化ループ・前提条件）

例：キーエンスが高収益 → ×「直販だから」で終わる。○「なぜ直販が"できる"のか、そして"なぜ他のメーカーは直販に切り替えられないのか"（既存代理店との関係・長年かけた営業組織・高粗利が高給を生み人材を集める循環…）」まで連れて行く。
どのラジオでも同じ。各トピックで「で、なぜ？」「じゃあ、なんで他はそうしない？」を2〜3回重ね、聞き手が"人に説明できる構造"を持ち帰れるようにする。

構成（article_playbook §1 を音声向けに適応）：
```
導入（話題性フック・事実ベース。kotaの実体験は捏造しない）
 → まず、○○とは（正体・現在地の数字を最小限）＋「○○を3つ共有します」予告
[STING]
 → トピック1（「実は〜」or「ポイント1」具体タイトル）
[STING]
 → トピック2
[STING]
 → トピック3 ＋ おわりに（小ネタ1個・余韻）
```
- **トピックの境目に `[STING]` だけの行を入れる**（produce.py がそこに区切り音を入れる）。3トピックなら [STING] は3つ。各トピックは深掘りの作法で厚く（1トピックあたり1000字前後）。
- やらない（playbook §2）：抽象化・補助線・系譜論／締めの断定キャッチコピー／独立した小ネタ集／会社情報や受賞歴の羅列／題材に合わないフレーミング／空振りの数値フック。
- **各回は自己完結（初めて聴く人前提）**：過去回を聴いている前提にしない。「前回」「このまえ○○の話をしました」のような**他エピソードへの参照を入れない**（消した回・聴いていない回を前提にすると意味不明になる）。毎回、必要な前提はその回の中で一から説明する。
- 文体：ですます調・会話寄り（「〜なんですよね」「実はこうなってて」「ちなみに」）。演説調は避ける。
- ⚠️ **語り手（AI）は、自分の個人的な体験・感想を捏造しない**（kota 指摘 2026-06-20）。「僕も観たんですが」「僕は〜と感じた」「先日行ってみたら」のような、視聴・体験・私的な意見を語り手のものとして語らない。あくまで事実を伝える解説者の立場。感想・解釈は出典に帰属させる（「〜と評されています」「〜と言われています」）。会話寄りの口調（〜なんですよね）はOKだが、それは口調であって、体験の捏造ではない。kota の実体験フックも捏造しない（事実ベースの話題性フックで入る）。
- 冒頭・結びの定型は各 `<SHOW>_ROUTINE.md` に従う（無ければ「こんにちは。○○ラジオ、今日のテーマは『△△』です。」「以上、今日の『△△』でした。これで明日、ちょっと語れますね。」）。

## STEP G. 読み上げ用カナ台本（Claude本人がふりがな・必須）
- 漢字を文脈に合う正しい読みでカナ化（固有名詞・人名・新語は WebSearch で読み確認）。**数字・日付・金額・件数は数字のまま**（synth が保護）。英略語・記号はカナ/言葉に。
- ⚠️ **固有名詞・キャラ名・人名で読みが自明でないものは、必ず調べてから書く**（kota 指摘 2026-06-19「ダー子を"だーし"と読んでいた」）。synth の形態素解析器は、こういう名前を平気で誤読する（例：ダー子→ダーシ、赤星→アカホシ）。**推測で漢字のまま渡さない**。対策＝①読みを WebSearch で確認し、②その語を読みカナで直接書く（「ダーこ」）か、③`data/yomi_dict.json` に登録する（全番組共通・kota に聞かず追記可。例 `"ダー子":"ダーこ"`）。少しでも「この読みで合ってる？」と思ったら調べる。
- **読点（、）は、人間がふつうに書くときと同じ自然な位置にだけ打つ**。＝文のあたまの接続語（「まず、」「でも、」「じつは、」「つまり、」）のあと、節と節の切れめ（〜て、／〜が、／〜ので、）、列挙の区切り、誤読を防ぎたいところ。**文節ごとに機械的に「、」を打たない**（kota 指摘 2026-06-18「文節全部に読点をつけて聞きづらい」。機械的な後処理での間引きは不自然になるので禁止＝以前あった `thin_commas` は撤去済み）。読点を打ちすぎた台本を出さないこと。声に出して読んで不自然な間がないか確認する。
- 自然な読点だと**同じ字数でも音声はやや短く**なる。9〜11分を狙うなら**カナ台本で約4000〜4800字**を目安に。生成後に必ず長さを確認。
- `[STING]` 行はそのまま残す（produce.py が解釈する）。
- 保存先：`scripts/_today_<SHOW>.txt`。毎回出る固有名詞は `data/yomi_dict.json` に追記（全番組共通・kotaに聞かず追記可）。

## STEP G-2. ファクトチェック＆批評ブラッシュアップ（音声化の前に必須・kota指示 2026-06-20）

**台本ができたら、produce する前に、必ず次の2つのレビューを通してブラッシュアップする。** スケジュール実行でも、手動でも、どの番組でも、毎回必ず行う（軽い回でも最低1パス）。

1. **ファクトチェック**（general-purpose サブエージェント1体）：台本テキストを渡し、「事実主張（固有名詞・人名・年号・数値・受賞/実績・因果関係・引用）を、信頼できる出典に照らして1件ずつ検証し、①誤り ②怪しい/裏が取れない ③ニュアンスが不正確、を列挙して」と指示。**戻ってきた指摘で台本を直す**（誤りは修正、裏が取れないものは外す/ぼかす、断定しすぎは緩める）。固有名詞の読みもここで最終確認（STEP G の読みルール）。
2. **批評ブラッシュアップ**（別の general-purpose サブエージェント1体）：台本を渡し、「面白さ・分かりやすさ・深掘りの作法（なぜ可能か/なぜ他はできないか）・話の流れ・つかみ・締め・冗長な繰り返し・空振りの数値フック、の観点で批評し、具体的な改善案を出して」と指示。**戻ってきた改善案で台本を磨く**（弱いトピックの差し替え、冗長削減、つかみと締めの強化）。
3. 修正後、**もう一度ファクトチェックで新たに足した部分に誤りがないか確認**し、長さも再確認。これらを通してから STEP H へ。

> 並列で2体同時に投げてよい（ファクトチェックと批評を同時に）。指摘がゼロでも「通した」と完了報告に1行残す。kota の評価が割れた作品・賛否のある題材は、断定を避け両論を正確に。

## STEP H. 音声合成（音楽入り）
```
python scripts/produce.py scripts/_today_<SHOW>.txt <SHOW>/episodes/<DATE>.mp3
```
- produce.py が：各セグメントを edge-tts（ナナミ固定）で合成 → **オープニング/エンディングのジングル＋[STING]位置に区切り音**を ffmpeg で合体。
- 音源は `assets/jingle.*` `assets/sting.*`（make_jingle.py 生成。SUNO製に差し替え可）。

## STEP I. メタ情報（日本語を引数で渡さない）
- `scripts/_meta_<SHOW>.json` を UTF-8 で書く：`{ "title": "...", "desc": "1〜2文。3トピックの要点＋主な出典名を入れると説明欄が充実" }`

## STEP J. 配信
```
python scripts/publish.py --show <SHOW> <SHOW>/episodes/<DATE>.mp3 <DATE>
```
- ⚠️ `--show <SHOW>` を必ず付ける（忘れると別番組のfeedを壊す）。

## STEP K. history 追記
- `data/<SHOW>_history.json` に今日のテーマを追記（date / theme / keywords）。STEP L の前に。

## STEP L. git push
```
git add -A && git commit -m "<SHOW> <DATE>: <テーマ名>" && git push
```

## 失敗時
- 詰まったら、その時点までをログに残して中断。不完全・壊れた音声は配信しない（push しない）。edge-tts/ffmpeg が落ちたら時間を置いて1回リトライ。

## 購読（kota 用メモ）
- Apple ポッドキャスト「ライブラリ→右上→番組をURLでフォロー」に **`<feed>?v=2` を貼る**（例 `https://kotayamanaka.github.io/kota-news-podcast/<SHOW>.xml?v=2`）。新規フィードは末尾 `?v=N` を付けないと「サーバまたは配信が見つからない」になる（Appleキャッシュ回避）。ダメなら `?v=3`。

## 完了報告
- 選んだテーマと「なぜ」を1行、トピック3つの見出し、音声の長さ。
